````markdown
Vì vậy, hệ thống cần chuyển sang xử lý bất đồng bộ. API chính chỉ nên ghi nhận yêu cầu gửi thông báo, sau đó đẩy công việc sang hàng đợi hoặc worker xử lý nền.

# 2. Các phương án kiến trúc

## Phương án 1: Spring Boot @Async + ThreadPoolTaskExecutor

### Cách hoạt động

API nhận yêu cầu gửi thông báo, sau đó gọi service có annotation `@Async`. Spring Boot sẽ đưa tác vụ gửi email/SMS sang một thread khác để xử lý nền.

### Ưu điểm

- Dễ triển khai.
- Phù hợp với hệ thống nhỏ.
- Không cần cài thêm message broker.
- Code đơn giản, dễ hiểu.

### Nhược điểm

- Không phù hợp với tải rất lớn như 1 triệu người dùng.
- Nếu server restart, các task đang chạy có thể bị mất.
- Khó retry bền vững.
- Khó kiểm soát rate limit.
- Khó scale ngang vì task nằm trong bộ nhớ của từng instance.
- Nếu thread pool bị đầy, hệ thống vẫn có thể bị nghẽn.

### Trường hợp nên dùng

Phù hợp cho hệ thống nhỏ, lượng thông báo ít, yêu cầu độ tin cậy không quá cao.

---

## Phương án 2: Message Queue với RabbitMQ

### Cách hoạt động

API nhận request tạo chiến dịch gửi thông báo, sau đó đẩy từng message hoặc từng batch message vào RabbitMQ. Các worker service sẽ consume message từ queue và gửi Email/SMS.

### Ưu điểm

- Tách API chính khỏi quá trình gửi thông báo.
- Có thể retry khi gửi thất bại.
- Có thể dùng Dead Letter Queue để lưu message lỗi.
- Dễ scale worker theo số lượng message.
- Phù hợp với workload dạng task/job.
- Kiểm soát được tốc độ consume để tránh vượt rate limit.

### Nhược điểm

- Cần vận hành thêm RabbitMQ.
- Cần thiết kế retry, DLQ, idempotency.
- Nếu số lượng message cực lớn, cần cấu hình broker cẩn thận.
- Không phù hợp bằng Kafka nếu cần event streaming khối lượng cực lớn.

### Trường hợp nên dùng

Phù hợp với hệ thống Notification Service cần xử lý bất đồng bộ đáng tin cậy, có retry, có kiểm soát tốc độ gửi, quy mô từ vừa đến lớn.

---

## Phương án 3: Event Streaming với Kafka

### Cách hoạt động

API hoặc Campaign Service publish event vào Kafka topic, ví dụ `promotion.notification.created`. Các consumer group sẽ đọc event, chia tải theo partition và gửi thông báo.

### Ưu điểm

- Khả năng chịu tải rất cao.
- Phù hợp khi số lượng message cực lớn.
- Dễ scale consumer theo partition.
- Có khả năng lưu lại event log.
- Phù hợp kiến trúc event-driven microservices.
- Có thể replay event khi cần xử lý lại.

### Nhược điểm

- Phức tạp hơn RabbitMQ.
- Cần hiểu rõ partition, offset, consumer group.
- Retry và DLQ cần tự thiết kế kỹ.
- Chi phí vận hành cao hơn.
- Không cần thiết nếu hệ thống chưa đến quy mô rất lớn.

### Trường hợp nên dùng

Phù hợp khi hệ thống có tải rất lớn, nhiều service cùng quan tâm đến event, cần event streaming và khả năng scale cao.

---

## Phương án 4: Batch Processing / Scheduled Job

### Cách hoạt động

Hệ thống lưu danh sách thông báo cần gửi vào database. Một scheduled job chạy định kỳ, lấy từng batch người dùng để gửi Email/SMS.

### Ưu điểm

- Dễ kiểm soát tiến độ gửi.
- Dễ giới hạn tốc độ gửi.
- Phù hợp với các chiến dịch không cần gửi ngay lập tức.
- Có thể lưu trạng thái từng bản ghi: PENDING, SENT, FAILED.

### Nhược điểm

- Không real-time.
- Nếu batch quá lớn, database có thể bị tải nặng.
- Cần thiết kế phân trang, lock, retry cẩn thận.
- Không linh hoạt bằng message queue khi cần scale lớn.

### Trường hợp nên dùng

Phù hợp với chiến dịch marketing có thể gửi chậm theo từng đợt, không yêu cầu gửi tức thời.

# 4. Phân tích What-if Scenario

## Tình huống 1: Nếu số người dùng tăng từ 1 triệu lên 10 triệu

Nếu số người dùng tăng lên 10 triệu, phương án `@Async` gần như không còn phù hợp vì task nằm trong bộ nhớ application và không có khả năng điều phối tải tốt.

Phương án phù hợp hơn là Kafka hoặc RabbitMQ kết hợp worker service độc lập.

- Nếu hệ thống cần event streaming lớn, nhiều consumer khác nhau, nên chọn Kafka.
- Nếu hệ thống chủ yếu là xử lý job gửi thông báo, RabbitMQ vẫn là lựa chọn tốt và dễ triển khai hơn Kafka.

## Tình huống 2: Nếu SMS provider giới hạn 1.000 request/phút

Cần áp dụng rate limiting ở worker.

Cách xử lý:

- Giới hạn số message consume mỗi phút.
- Dùng token bucket hoặc fixed window rate limiter.
- Tách queue Email và SMS riêng.
- Với SMS, worker chỉ gửi tối đa 1.000 request/phút.
- Message còn lại tiếp tục nằm trong queue.

## Tình huống 3: Nếu một phần message gửi thất bại

Cần có cơ chế retry.

Ví dụ:

- Retry lần 1 sau 1 phút.
- Retry lần 2 sau 5 phút.
- Retry lần 3 sau 15 phút.
- Nếu vẫn lỗi, đưa vào Dead Letter Queue.

Để tránh gửi trùng, cần thiết kế idempotency key:

```text
campaignId + userId + channel
```
````

Trước khi gửi, worker kiểm tra xem thông báo này đã được gửi thành công chưa. Nếu đã gửi rồi thì bỏ qua.

## Tình huống 4: Nếu Email/SMS provider bị downtime

Cần xử lý fallback:

- Không xóa message khỏi queue nếu gửi thất bại.
- Retry với backoff.
- Nếu lỗi kéo dài, đưa message vào DLQ.
- Có thể chuyển sang provider dự phòng.
- Cảnh báo qua monitoring như Prometheus, Grafana, ELK hoặc hệ thống alert.

## Tình huống 5: Nếu nhiều chiến dịch khuyến mãi chạy cùng lúc

Cần tránh để một chiến dịch lớn chiếm hết tài nguyên.

Giải pháp:

- Tách queue theo priority.
- Có rate limit theo campaign.
- Có quota gửi theo từng loại thông báo.
- Worker lấy message công bằng giữa các campaign.
- Lưu trạng thái campaign để tạm dừng hoặc tiếp tục gửi khi cần.

# 5. Kiến trúc đề xuất cuối cùng

Với bài toán gửi Email/SMS cho 1 triệu người dùng, phương án tối ưu nên là:

## Spring Boot + RabbitMQ + Worker Service + Retry + Dead Letter Queue + Idempotency

### Lý do chọn RabbitMQ

RabbitMQ phù hợp vì bài toán này thiên về xử lý job gửi thông báo hơn là event streaming phức tạp. So với `@Async`, RabbitMQ đáng tin cậy và dễ scale hơn. So với Kafka, RabbitMQ đơn giản hơn, dễ triển khai hơn nhưng vẫn đáp ứng tốt nhu cầu gửi thông báo cho quy mô lớn.

### Luồng xử lý đề xuất

1. Admin tạo chiến dịch khuyến mãi.
2. Campaign Service lấy danh sách user cần gửi.
3. Campaign Service chia danh sách thành nhiều message hoặc batch nhỏ.
4. Đẩy message vào RabbitMQ.
5. Notification Worker consume message.
6. Worker kiểm tra idempotency để tránh gửi trùng.
7. Worker gửi Email/SMS qua provider.
8. Nếu thành công, lưu trạng thái SENT.
9. Nếu thất bại, retry theo backoff.
10. Nếu vượt số lần retry, đưa vào Dead Letter Queue.

# 6. Skeleton Java/Spring Boot minh họa

```java
package com.example.notification.dto;

public record NotificationMessage(
        String campaignId,
        Long userId,
        String channel,
        String recipient,
        String title,
        String content
) {
    public String idempotencyKey() {
        return campaignId + ":" + userId + ":" + channel;
    }
}
```

```java
package com.example.notification.producer;

import com.example.notification.dto.NotificationMessage;
import lombok.RequiredArgsConstructor;
import org.springframework.amqp.rabbit.core.RabbitTemplate;
import org.springframework.stereotype.Service;

@Service
@RequiredArgsConstructor
public class NotificationProducer {

    private final RabbitTemplate rabbitTemplate;

    public void sendToQueue(NotificationMessage message) {
        rabbitTemplate.convertAndSend(
                "notification.exchange",
                "notification.send",
                message
        );
    }
}
```

```java
package com.example.notification.consumer;

import com.example.notification.dto.NotificationMessage;
import com.example.notification.service.IdempotencyService;
import com.example.notification.service.NotificationSender;
import lombok.RequiredArgsConstructor;
import org.springframework.amqp.rabbit.annotation.RabbitListener;
import org.springframework.stereotype.Component;

@Component
@RequiredArgsConstructor
public class NotificationConsumer {

    private final NotificationSender notificationSender;
    private final IdempotencyService idempotencyService;

    @RabbitListener(queues = "notification.queue")
    public void consume(NotificationMessage message) {
        String key = message.idempotencyKey();

        if (idempotencyService.alreadyProcessed(key)) {
            return;
        }

        notificationSender.send(message);

        idempotencyService.markAsProcessed(key);
    }
}
```

```java
package com.example.notification.service;

import com.example.notification.dto.NotificationMessage;
import org.springframework.stereotype.Service;

@Service
public class NotificationSender {

    public void send(NotificationMessage message) {
        if ("EMAIL".equalsIgnoreCase(message.channel())) {
            sendEmail(message);
        } else if ("SMS".equalsIgnoreCase(message.channel())) {
            sendSms(message);
        } else {
            throw new IllegalArgumentException("Unsupported channel: " + message.channel());
        }
    }

    private void sendEmail(NotificationMessage message) {
        // Gọi Email Provider tại đây
        System.out.println("Sending email to: " + message.recipient());
    }

    private void sendSms(NotificationMessage message) {
        // Gọi SMS Provider tại đây
        System.out.println("Sending SMS to: " + message.recipient());
    }
}
```

```java
package com.example.notification.service;

import org.springframework.stereotype.Service;

import java.util.Set;
import java.util.concurrent.ConcurrentHashMap;

@Service
public class IdempotencyService {

    private final Set<String> processedKeys = ConcurrentHashMap.newKeySet();

    public boolean alreadyProcessed(String key) {
        return processedKeys.contains(key);
    }

    public void markAsProcessed(String key) {
        processedKeys.add(key);
    }
}
```

```java
package com.example.notification.config;

import org.springframework.amqp.core.*;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class RabbitMQConfig {

    public static final String EXCHANGE = "notification.exchange";
    public static final String QUEUE = "notification.queue";
    public static final String DLQ = "notification.dlq";
    public static final String ROUTING_KEY = "notification.send";

    @Bean
    public DirectExchange notificationExchange() {
        return new DirectExchange(EXCHANGE);
    }

    @Bean
    public Queue notificationQueue() {
        return QueueBuilder.durable(QUEUE)
                .deadLetterExchange(EXCHANGE)
                .deadLetterRoutingKey("notification.dead")
                .build();
    }

    @Bean
    public Queue deadLetterQueue() {
        return QueueBuilder.durable(DLQ).build();
    }

    @Bean
    public Binding notificationBinding() {
        return BindingBuilder
                .bind(notificationQueue())
                .to(notificationExchange())
                .with(ROUTING_KEY);
    }

    @Bean
    public Binding deadLetterBinding() {
        return BindingBuilder
                .bind(deadLetterQueue())
                .to(notificationExchange())
                .with("notification.dead");
    }
}
```

# 7. Kết luận

Không nên chỉ dùng `@Async` cho bài toán gửi thông báo tới 1 triệu người dùng vì nó thiếu độ tin cậy, khó retry, khó scale và dễ mất task khi server gặp sự cố.

Giải pháp phù hợp hơn là thiết kế bất đồng bộ dựa trên Message Queue, cụ thể là RabbitMQ hoặc Kafka. Với bài toán Notification Service dạng gửi job Email/SMS, RabbitMQ là lựa chọn cân bằng nhất vì vừa đủ mạnh cho production, vừa dễ triển khai hơn Kafka.

Kiến trúc đề xuất:

```text
Campaign Service
      |
      v
RabbitMQ Exchange
      |
      v
Notification Queue
      |
      v
Notification Worker
      |
      v
Email/SMS Provider

Kèm theo:
- Retry
- Dead Letter Queue
- Idempotency Key
- Rate Limiting
- Monitoring
```
